# Context Health Check

当用户触发此 skill 时，执行以下评估并输出报告。

## Token 估算方法

LLM 无法直接读取已消耗 token 数量，使用以下代理公式估算：

### 估算公式

```
已消耗 ≈ 系统开销 + Σ(每轮消耗)

系统开销 ≈ 15K-25K tokens（系统提示 + skills 元数据 + 工作区信息）

每轮消耗 ≈ 用户消息(~200-500) 
           + 工具调用结果(每次 ~500-5000，大文件读取 ~3000-8000)
           + 助手回复(~500-2000)
```

### 高消耗操作参考值

| 操作 | 估算 tokens |
|------|:----------:|
| 读取 <100 行文件 | ~500-1500 |
| 读取 200-500 行文件 | ~3000-5000 |
| 读取 500+ 行文件（如 STATUS.md） | ~5000-10000 |
| grep 搜索结果（50条） | ~3000-5000 |
| 长命令输出 | ~2000-8000 |
| 生成/编辑一个文件 | ~1000-3000 |
| 浏览器操作（含 DOM） | ~5000-15000 |

### 模型上下文窗口

| 模型 | 总窗口 | 可用（扣除系统开销） | 安全阈值(80%) |
|------|:------:|:------------------:|:------------:|
| Claude Opus/Sonnet (Anthropic) | 200K | ~175K | **140K** |
| GPT-4o (OpenAI) | 128K | ~105K | **84K** |
| Gemini Pro (Google) | 1M+ | ~950K+ | **不太会爆** |

⚠️ **必须先读 `USER.md` 中的 `模型偏好` 字段**。如找到则使用对应模型的窗口大小，未找到则默认 Claude 200K。**禁止默认使用 Gemini 1M 或其他模型。**

## 评估流程

1. **识别当前模型**（读取 `USER.md` → `模型偏好` 字段。未找到则默认 Claude 200K。**不要猜测，不要用 Gemini 1M**）
2. **计算对话轮次**（user↔assistant 交互次数）
3. **估算工具调用 token 消耗**（按上表逐项累加）
4. **计算总消耗估值**
5. **计算剩余百分比** = (可用窗口 - 已消耗) / 可用窗口
6. **给出健康等级**

## 健康等级

| 等级 | 剩余百分比 | 建议 |
|:---:|:---------:|------|
| 🟢 **充裕** | >50% 剩余 | 继续工作，无需担心 |
| 🟡 **注意** | 20-50% 剩余 | 完成当前任务就开新对话。避免加载更多大文件。避免长输出命令 |
| 🔴 **危险** | <20% 剩余 | ⚠️ 立即更新状态文件并开新对话。继续工作大概率会出现遗忘/重复/质量下降 |

### 快速判断经验法则（不想算的时候）

Claude 200K 窗口下：
- **≤5 轮 + 1个大文件** → 🟢 
- **10-15 轮 + 2-3 个大文件 + 多次代码编辑** → 🟡
- **>18 轮** 或 **>5 个大文件读取** 或 **开始遗忘早期内容** → 🔴

## 输出格式

```
## 🧠 上下文健康检查

**模型**: [Claude/GPT-4o/Gemini/...]（上下文窗口 XK）

| 指标 | 当前值 | 估算 tokens |
|------|:-----:|:----------:|
| 系统开销 | — | ~20K |
| 对话轮次 | X 轮 | ~YK |
| 大文件读取 | [列出文件名+行数] | ~ZK |
| 工具调用 | ~N 次 | ~WK |
| **总消耗估算** | — | **~TK** |
| **剩余估算** | — | **~RK (P%)** |

**健康等级**: 🟢/🟡/🔴

**建议**: [具体建议]
**预计还能支撑**: ~M 轮普通对话 / ~L 次大文件读取
```

## 防御式自动保存（Defensive Auto-Save）

> **核心问题**: 🔴 检测依赖 Agent 有机会执行——突然爆掉时 Agent 没有机会写任何东西。
> **解决方案**: 不等 🔴，每 10 轮交互主动写一次。

### 规则

1. **每 10 轮交互**（user↔assistant 来回 = 1 轮），Agent **必须**静默更新 `memory/CRASH_CONTEXT.md`
2. **静默 = 不打断用户**。不需要告知、不需要确认。在回复用户前或工具调用间隙完成
3. **增量更新**：只更新变化的字段，不全量重写
4. **格式**与 🔴 时写入的相同（见下方附加规则 #1）
5. **恢复后删除**：`/resume crash` 成功恢复后，删除 `CRASH_CONTEXT.md`（避免过时数据）
6. **触发时机检查表**：
   - 第 10 轮 → 首次写入 ✅
   - 第 20 轮 → 更新 ✅
   - 第 30 轮 → 更新 ✅（此时通常已接近 🟡/🔴）
   - 🔴 检测触发 → 最终更新（含「建议下一步」）✅

### 防御层级

| 层级 | 触发条件 | 可靠性 |
|:----:|----------|:------:|
| **L0 防御** | 每 10 轮静默写入 | ⭐⭐⭐ 最可靠 |
| **L1 预警** | `/ctx` 检测到 🔴 时自动写入 | ⭐⭐ 需检测机会 |
| **L2 回退** | `/resume crash` 扫描 `convo/` _CRASH 文件 | ⭐ 兜底 |

## 附加规则

1. 🔴 危险时，**自动**执行 memory 快照 + crash 恢复文件（MUSE compact:before 协议）：
   - 将当前进度写入 `memory/YYYY-MM-DD.md`（不等用户 /bye）
   - 更新 `task.md` 进度标记
   - **写 `memory/CRASH_CONTEXT.md`**（结构化恢复文件，下次 `/resume crash` 自动读取）：
     ```markdown
     # Crash Context — YYYY-MM-DD HH:MM
     ## 当前角色: [strategy/build/growth/qa]
     ## 当前任务: [正在做什么]
     ## 未完成: [列表]
     ## 关键决策: [本轮做的决策]
     ## 用户最后的问题: [如有]
     ## 建议下一步: [恢复后应该做什么]
     ```
   - 然后提醒用户：
     ```
     ⚠️ 上下文即将耗尽！已自动保存:
     - memory/YYYY-MM-DD.md ✅
     - memory/CRASH_CONTEXT.md ✅
     请结束对话。新对话用 `/resume crash` 自动恢复。
     如需导出本次对话，保存到: convo/YYMMDD/YYMMDD-NN-desc_CRASH.md
     ```
2. 如果当前对话还没加载任何状态文件且 ≤3 轮，直接报 🟢 不需要详细计算
3. 自然语言触发也生效（"还能聊多久？"、"context够吗"等）
4. 如果检测到自己开始遗忘对话早期内容或重复自己说过的话，即使按公式算还有空间，也直接报 🔴
